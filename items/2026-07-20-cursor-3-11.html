Contexte étendu

Cursor 3.11 (10 juil.) est une release d'ergonomie agentique. Les side chats répondent à un problème concret des longues sessions agent : explorer une question annexe sans polluer ni casser la boucle principale. Ouverts par /side, /btw ou le bouton +, ils branchent depuis le thread principal en héritant de son contexte, et leur résultat se re-injecte par at-mention. La recherche de transcripts d'agents ajoute un index local exposé via Cmd+K dans l'Agents Window, dimensionné pour des milliers de conversations, avec Cmd+F intra-conversation — une réponse directe à l'accumulation de sessions agent qu'aucun IDE ne gérait proprement.

La brique la plus structurante est Team MCP servers : un administrateur configure un serveur MCP une seule fois et le distribue vers les cloud agents, l'Agents Window, l'IDE et le CLI, avec un marketplace d'équipe (Dashboard → Integrations & MCP) où les membres installent localement les intégrations approuvées sans reconfigurer chaque serveur. C'est le prolongement de la 3.10 (30 juin), qui avait introduit les Team MCPs et les org groups.

S'y ajoutent cinq nouveaux hook events pour cloud agents et des pickers projet/repo redessinés. Ces hooks rapprochent Cursor du modèle d'automation en arrière-plan que Claude Code industrialise déjà (subagents de fond, auto-commit/push/PR draft), et confirment que la bataille entre IDE agentiques se joue moins sur le modèle sous-jacent que sur l'orchestration d'équipe et la gouvernance des connecteurs.

🎯 Pourquoi c'est important pour Lynxter

Le pattern Team MCP servers est directement transposable chez Lynxter. Plutôt que de faire configurer à chaque poste dev/support les connecteurs MCP maison — accès à la doc technique S300X/S600D, base de tickets support, référentiel de profils d'impression — un administrateur les définit une fois et les pousse vers IDE, CLI et agents cloud via un marketplace interne. Cela élimine la dérive de configuration (versions de serveurs MCP divergentes entre postes) et pose une couche de gouvernance : seuls les connecteurs approuvés sont installables.

C'est le même besoin que Claude Code adresse côté Anthropic, et le bon moment pour trancher : si la stack Lynxter est majoritairement Claude Code/Cowork, réplique ce pattern avec les mécanismes MCP natifs de Claude Code ; si des devs utilisent Cursor en parallèle, la distribution Team MCP de la 3.11 permet d'aligner les deux environnements sur le même jeu de connecteurs approuvés. À tester cette semaine : centraliser un premier connecteur MCP « doc S300X/S600D » et mesurer le temps de setup gagné vs configuration par poste.
