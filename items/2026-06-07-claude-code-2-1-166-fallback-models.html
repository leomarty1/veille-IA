Contexte étendu

Entre le 5 et le 6 juin 2026, Claude Code passe de v2.1.165 à v2.1.168 — quatre builds rapprochés dont une release feature (166) et trois correctifs de fiabilité (165, 167, 168). L'apport central est le fallbackModel : un réglage qui accepte jusqu'à trois modèles de repli, essayés dans l'ordre quand le modèle primaire est overloaded ou indisponible. Nouveauté de cette version : --fallback-model s'applique désormais aussi aux sessions interactives, alors qu'il était jusqu'ici réservé aux exécutions non-interactives (CI, headless, Routines). Claude Code va plus loin en retentant une fois sur le modèle de fallback quand l'API rejette une erreur non-retryable inattendue — les erreurs d'auth, de rate-limit, de taille de requête et de transport, elles, remontent immédiatement sans bascule.

Le deuxième chantier est la granularité des règles de permission. Les deny rules supportent maintenant le glob en position nom-de-tool : "*" dans une deny rule bloque l'intégralité des tools, ce qui permet d'écrire une politique « tout interdit sauf liste blanche explicite ». Symétriquement, les allow rules rejettent les globs non-MCP (on ne peut pas auto-autoriser large via wildcard), et un nom de tool inconnu dans une deny rule déclenche désormais un warning au démarrage plutôt qu'un échec silencieux — fini les politiques qui ne s'appliquent pas parce qu'on a mal orthographié un nom de tool. Les managed settings invalides ne désactivent plus silencieusement l'enforcement des policies valides restantes.

Le troisième apport est sécuritaire et arrive à point nommé après le lancement des Dynamic Workflows (jusqu'à 1 000 sous-agents) la semaine précédente : durcissement de la messagerie cross-session. Les messages relayés via SendMessage depuis d'autres sessions Claude ne portent plus l'autorité de l'utilisateur — les sessions réceptrices refusent les demandes de permission relayées, et le mode auto les bloque purement. C'est une mitigation directe contre l'escalade de privilèges inter-agents : un sous-agent ne peut plus faire approuver une action sensible en la relayant à une session qui, elle, tournerait en mode permissif. Côté correctifs notables : MAX_THINKING_TOKENS=0 et --thinking disabled désactivent réellement le thinking sur les modèles qui pensent par défaut via l'API Claude (les providers tiers restent inchangés) ; fixes du flicker terminal JetBrains sur 2026.1+, des caractères Shift+non-ASCII perdus sous protocole Kitty (WezTerm, Ghostty, kitty), du hang de validation PowerShell sur Windows, et des process claude --bg-pty-host orphelins coincés à 100 % CPU sur macOS.

🎯 Pourquoi c'est important pour Lynxter

Le fallbackModel élimine la classe d'échecs la plus pénible pour tes Routines : la saturation API. La routine veille-IA tourne le lundi à 01:00 Europe/Paris, soit dimanche soir aux US — pas l'heure de pointe, mais les pics de charge Anthropic sont devenus fréquents depuis l'explosion de l'adoption (run-rate 47 Md$). Aujourd'hui, un brief peut planter sur un model overloaded en plein milieu de 30 WebSearch. Avec fallbackModel: ["claude-opus-4-8", "claude-sonnet-4-6", "claude-haiku-4-5"] dans le .claude/settings.json du repo, la routine bascule automatiquement sur Sonnet puis Haiku plutôt que d'échouer — quitte à dégrader légèrement la qualité de synthèse, ce qui est toujours préférable à un run sans brief publié. À faire cette semaine : ajouter le réglage à la config partagée et tester un run forcé en simulant l'indisponibilité du primaire.

Les globs dans les deny rules débloquent une posture « deny-all par défaut » pour les Routines autonomes. Jusqu'ici, sécuriser une Routine qui tourne sans humain dans la boucle (génération de brief, scan de tickets, audit firmware) imposait d'énumérer chaque tool à bloquer. Avec "*" en deny + une allowlist explicite des seuls tools nécessaires (WebSearch, WebFetch, Read, le MCP GitHub), tu obtiens une politique minimale-privilège propre, et le warning au démarrage t'alerte si un nom de tool est mal écrit. Pour les Routines qui touchent au repo firmware S300X/S600D, c'est exactement le garde-fou qu'il faut : autoriser la lecture et l'analyse, interdire par défaut toute écriture ou exécution shell non explicitement whitelistée.

Le durcissement cross-session est la pièce manquante pour exploiter les Dynamic Workflows en confiance. La semaine dernière, on notait que les workflows multi-agents (1 000 sous-agents max) étaient prometteurs pour paralléliser la veille et les audits firmware. Le risque latent était la confusion d'autorité entre agents. Ce fix le ferme : un sous-agent compromis (par une injection de prompt dans une page web fetchée, par exemple) ne peut plus escalader en relayant une demande de permission à une session en mode auto. Pour Lynxter, cela rend acceptable de lancer des workflows qui fetchent du contenu externe non fiable (forums, docs fournisseurs, issues GitHub tierces) sans craindre qu'un agent piégé n'obtienne des droits d'écriture. Comparaison : Cursor 3.6 traitait le même problème côté approbation avec un classifier subagent (autoriser/retenter/demander) ; Anthropic répond par une frontière d'autorité dure entre sessions — approche plus restrictive, mais plus auditable.
