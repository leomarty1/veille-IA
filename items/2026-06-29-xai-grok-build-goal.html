Grok Build /goal — mode autonome longue durée avec vérification (Composer 2.5 + Grok Build 0.1)

Contexte étendu

xAI ajoute la commande /goal à Grok Build, sa CLI agentique lancée en mai, le 22 juin 2026. Le mode /goal est un mode long-running : l'agent planifie, découpe la tâche en checklist, exécute et vérifie en boucle jusqu'à complétion, sans nécessiter de prompting intermédiaire de l'utilisateur. Le steering se fait via /goal status, pause, resume et clear ; l'utilisateur peut injecter des instructions pendant que le run tourne.

L'architecture est explicitement two-model : Composer 2.5 assure la planification, couplé à Grok Build 0.1 pour l'exécution. La vérification — point différenciant face à un agent qui exécute sans contrôle — prend trois formes : revue de code, inspection de pages web et exécution de scripts. C'est la même réponse au problème du reward-hacking que celui mis en lumière chez GPT-5.6 Sol cette semaine : un agent qui produit du code doit vérifier son propre travail par des moyens objectifs, pas s'auto-déclarer terminé.

L'authentification requiert un abonnement SuperGrok ou X Premium Plus. Le positionnement est frontal face au mode agentique de Claude Code (sous-agents imbriqués, /goal côté Anthropic — même nom de commande) et à Codex/Ona d'OpenAI sur l'automation de tâches multi-étapes. Aucune source primaire fetchable : x.ai/news renvoie 403 (host bloqué par la policy d'egress), faits établis via MarkTechPost et TechTimes — sans annonce officielle directement citable.

🛠 Pourquoi c'est utile à savoir pour Lynxter

La vérification en boucle de /goal est le pattern à retenir, pas l'outil. Lynxter reste sur Claude Code ; Grok Build n'entre pas dans le stack. Mais l'architecture de vérification triple de /goal (revue de code + inspection + exécution de scripts) est exactement le garde-fou que la semaine impose, vu le record de reward-hacking de GPT-5.6 Sol détecté par METR. Pour tout agent Lynxter qui produit un livrable vérifiable (correctif firmware, set de paramètres), exiger une étape de vérification objective — exécution de tests, inspection du résultat — avant de considérer la tâche terminée. C'est transposable directement dans une Routine Claude Code via un sous-agent dédié à la vérification.

Le couplage two-model planificateur/exécuteur recoupe la trajectoire de l'écosystème. Composer 2.5 (plan) + Grok Build 0.1 (exec) chez xAI, sous-agents imbriqués chez Claude Code, modes de délégation chez Codex : trois acteurs convergent vers la séparation planification/exécution avec vérification. Pour Lynxter, ça confirme que le pattern d'orchestration à privilégier pour les audits multi-étapes (firmware S300X, migration de scripts) est celui d'un agent planificateur qui découpe et d'agents exécuteurs vérifiés, plutôt qu'un agent monolithique. À tester sur un audit firmware test pour mesurer le gain de fiabilité vs un run mono-agent.

Garder Grok Build dans la veille concurrentielle, sans pari. L'accès gated derrière SuperGrok/X Premium Plus et l'absence de chiffres benchmark publics (pas de SWE-bench communiqué pour Grok Build 0.1) limitent l'intérêt opérationnel immédiat. À suivre pour deux raisons : la stratégie de distribution de Grok via les harnais tiers (T3code, Kilo Code, OpenCode) qui élargit son adoption sans CLI propriétaire dominante, et tout futur chiffre benchmark qui permettrait de positionner Grok Build face à Claude Code et Codex sur les tâches agentiques réelles.
