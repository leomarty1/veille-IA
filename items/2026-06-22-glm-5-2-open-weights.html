GLM-5.2 (Z.ai) — open weights MIT qui passe GPT-5.5 sur le code à 1/6 du prix

Contexte étendu

Z.ai publie le 16 juin GLM-5.2, son modèle phare pour les tâches longue durée, en open weights sous licence MIT — poids disponibles sur Hugging Face (zai-org/GLM-5.2) et ModelScope. C'est un Mixture-of-Experts d'environ 750 milliards de paramètres au total, ~40 milliards actifs par token, avec une fenêtre de contexte de 1 million de tokens. Les poids complets pèsent 1,51 To en BF16, ce qui situe le ticket d'entrée self-host autour de 8× H100 ; l'inférence est supportée par transformers, vLLM, SGLang et d'autres frameworks.

Sur les benchmarks de code, GLM-5.2 dépasse GPT-5.5 sur plusieurs tâches longue durée : SWE-bench Pro à 62.1 % (contre 58.6 % pour GPT-5.5) et FrontierSWE à 74.4 % (contre 72.6 %), un score qui talonne Opus 4.8 (75.1 %) — du jamais vu pour un modèle open weights sur ce type de benchmark agentique. Côté coût, l'écart est l'argument massue : 1,40 $/M tokens en entrée et 4,40 $ en sortie, soit ~5,80 $ combinés, contre 5 $ / 30 $ (~35 $) pour GPT-5.5 — environ un sixième du prix pour une performance code comparable ou supérieure.

La nuance importante est double. D'abord le matériel : un modèle de ~750 Md de paramètres n'est pas un poids qu'on fait tourner sur une station de travail — le self-host suppose un cluster GPU (8× H100) ou une location cloud, ce qui déplace le coût du token vers le CapEx/OpEx infra. Ensuite la gouvernance des données : passer par l'API hébergée (en Chine) expose à un risque de juridiction sur les données envoyées, ce que plusieurs analyses ont souligné ; le self-host des poids ouverts neutralise ce risque et c'est précisément là que réside l'intérêt stratégique du MIT. GLM-5.2 confirme la tendance 2026 : l'open weights chinois ferme l'écart avec la frontière fermée sur le code, en cassant les prix.

🛠 Pourquoi c'est utile à savoir pour Lynxter

GLM-5.2 est la première option open weights crédible pour un plan B souverain sur le code. Pour les comptes défense/médical/aérospatial de Lynxter qui refusent d'envoyer code ou données process vers un LLM hébergé hors UE, un modèle MIT que l'on peut faire tourner entièrement on-premises et qui passe GPT-5.5 sur SWE-bench Pro change la conversation. C'est l'argument souveraineté de Mistral, mais avec un score de code frontière. À cadrer toutefois : c'est un plan B / un levier de négociation, pas un remplacement de Claude Code dans le workflow quotidien tant que l'écart d'outillage agentique (Artifacts, agent teams, MCP) reste en faveur d'Anthropic.

Le test à faire : benchmarker GLM-5.2 contre Claude sur tes tâches réelles, pas sur les benchmarks publics. Prends 2-3 tâches de code représentatives et non sensibles (refactor de scripts de calibration, génération de tests pour un profil firmware, parsing de logs gcode), lance-les via l'API GLM-5.2 et via Claude Code, et mesure l'écart de qualité sur ton terrain. Si le gap est faible, tu tiens un chiffrage de coût alternatif et une preuve à présenter aux comptes sensibles ; s'il est large, tu sais que la prime Claude est justifiée. N'utilise l'API hébergée que pour des tâches non confidentielles — réserve le self-host aux données sensibles.

Évalue le vrai coût total avant de promettre du self-host. Le « 1/6 du prix » est un coût par token API ; le self-host de ~750 Md de paramètres (8× H100, 1,51 To BF16) déplace la dépense vers l'infra et l'exploitation. Pour Lynxter, le calcul n'a de sens que si (a) un client souverain le finance, ou (b) le volume de tokens justifie l'amortissement GPU. Sinon, l'option pragmatique reste l'API pour le non-sensible + Claude pour le quotidien — en gardant GLM-5.2 documenté comme architecture de repli souveraine.
