Contexte étendu

xAI introduit Grok Skills le 18 mai 2026 sur Grok 4.3 — disponible sur grok.com web, iOS et Android. Concept : tu enseignes à Grok un workflow ou une préférence une seule fois (formatage de réponse, terminologie métier, structure de doc), et Grok les applique automatiquement dans toutes les conversations suivantes — y compris les conversations indépendantes lancées plus tard. C'est un mini-CLAUDE.md géré côté serveur, mais étendu au-delà du dev : préférences personnelles, règles de formatage, étapes spécifiques d'un workflow récurrent. xAI livre un pack initial intégré : génération de fichiers production-ready (Word, PowerPoint, Excel, PDF) avec un style maison, restructuration de fichiers uploadés (le user uploade un brouillon, Grok le polit/étend selon la skill active).

Différence notable avec le pattern Anthropic Skills (lancé fin 2025) : chez xAI les skills sont gérées côté grok.com/app, pas dans le repo local ou un système de fichiers. Plus simple côté UX, moins versionnable côté équipe. Le système d'override garantit que si tu crées ta propre skill avec le même nom qu'une skill xAI native, ta version prend priorité — pas de conflit silencieux. Skills live sur Grok 4.3 (le modèle cost-efficient à $1.25/M input, contexte 1M tokens, support vidéo natif).

En parallèle des Skills, xAI annonce de nouveaux connecteurs tiers : Vercel (déploiement de sites web depuis une conversation Grok), Canva (création de designs), Gamma (génération de présentations), S&P Global (données marché temps réel). Pattern lecture-écriture sur les apps tierces, comparable au catalogue connecteurs MCP Anthropic mais en système propriétaire xAI plutôt qu'en standard ouvert. Avant cette release, Grok n'avait pas de pattern persistance forte cross-conversations — c'est xAI qui rattrape un pattern produit que Anthropic, OpenAI (avec ChatGPT Memory) et même Gemini Memory ont déjà.

🛠 Pourquoi c'est utile pour Lynxter

Grok n'est pas un acteur Lynxter, mais l'arrivée de Skills consolide un pattern d'industrie utile à connaître. La vraie question est : ton CLAUDE.md actuel — qui contient déjà des skills implicites (style éditorial du brief, structure des items, profondeur par tag) — est-il portable à d'autres modèles si jamais tu veux multi-vendor ? La réponse est non aujourd'hui parce que CLAUDE.md est versionné dans le repo et chargé en contexte par Claude Code spécifiquement. Pour multi-vendor proof, il faudra penser tes prompts produit comme des "skills" portables : un fichier de spécification éditoriale + un fichier de structure HTML + un fichier de scoring, chacun consommable par Claude Code aujourd'hui, par Antigravity demain via son SDK, par Grok ou GPT-5.5 ailleurs.

Pas d'action immédiate, mais à garder en tête pour les Routines T3 2026 : si Lynxter envisage de tester Antigravity en parallèle (cf. l'item 🎯 sur Antigravity 2.0), refactorer la routine veille-IA en briques portables est la première étape — séparer la spec éditoriale (style, scoring, profondeur) du runtime (qui lit data.json, écrit les fichiers, push sur git). Une fois découplé, swap du runtime devient trivial et le benchmark multi-modèle aussi.
