Project Fetch Phase 2 — Opus 4.7 pilote un robodog ~20× plus vite qu'une équipe humaine

Contexte étendu

Project Fetch est une expérience de la Frontier Red Team d'Anthropic : en août 2025, des équipes humaines (avec et sans assistance Claude) avaient dû programmer un robodog du commerce pour accomplir une série de tâches de contrôle, jusqu'à aller chercher une balle. La Phase 2, publiée le 18 juin 2026, refait l'exercice en confiant le contrôle à Opus 4.7 en autonomie, qui écrit et itère le code de pilotage. Le résultat chiffré : sur chaque tâche complétée par au moins une équipe humaine en 2025, Opus 4.7 va au moins 10× plus vite, et globalement ~20× plus vite que la meilleure équipe humaine. Comparé aux deux équipes de référence, le modèle est ~37× plus rapide que l'équipe sans IA et ~19× plus rapide que l'équipe qui utilisait déjà un assistant Claude.

L'efficacité ne se lit pas qu'en vitesse : Opus 4.7 atteint un résultat égal ou supérieur aux deux équipes humaines en générant ~10× moins de code que l'équipe « Team Claude ». Autrement dit, le modèle ne brute-force pas en empilant les lignes — il produit un programme de contrôle plus compact. C'est cohérent avec ce qu'Anthropic observe par ailleurs sur les tâches d'ingénierie logicielle longues : la valeur n'est pas le volume produit mais la justesse de la structure.

La limite est tout aussi instructive que la performance. Opus 4.7 échoue à la tâche « fetch » — déplacer précisément la balle — parce qu'elle exige un contrôle en boucle fermée : percevoir, corriger le mouvement, re-percevoir, à haute fréquence. Anthropic est explicite : le modèle peine sur l'ajustement moteur adaptatif sur retour visuel temps réel, le type de compétence que les humains acquièrent par la pratique sensorimotrice. La leçon : les LLM actuels excellent à générer et structurer du code de contrôle à partir d'une spécification, mais ne remplacent pas une boucle d'asservissement temps réel. C'est une frontière de capacité, pas un détail d'implémentation.

🎯 Pourquoi c'est important pour Lynxter

Project Fetch est la cartographie exacte de ce qu'un agent peut et ne peut pas faire sur une machine Lynxter. Côté « peut » : générer du code de contrôle, des séquences de test, du parsing de logs, des profils de slicing, des scripts de calibration — là où Anthropic mesure 20× de vitesse et 10× moins de code. Côté « ne peut pas » : fermer la boucle capteur→actionneur en temps réel — asservissement de la tête, compensation thermique active, correction de trajectoire en cours d'impression. La S300X/S600D vit précisément dans cette boucle fermée. Conclusion opérationnelle : déploie les agents sur la couche logicielle/offline, garde le temps réel pour le firmware déterministe.

Le bon pilote d'usage : la génération de jeux de tests firmware. Plutôt que d'attendre un agent qui pilote la machine, fais générer par Claude Code une suite de tests pour un profil firmware S600D (cas limites de température, vitesses, rétractions) et compare le temps vs une écriture manuelle — l'ordre de grandeur Anthropic (20× plus vite, 10× moins de code) est ce que tu devrais approcher sur une tâche purement logicielle bien spécifiée. Si le gain se confirme, c'est un pattern reproductible pour toute la matrice de profils.

À surveiller sur 12-18 mois : la boucle fermée se rapprochera. Le point de blocage (contrôle moteur temps réel sur perception) est un axe de recherche actif ; le jour où un modèle frontière le franchit, le périmètre « agent qui pilote du hardware » s'ouvre pour de bon. Pour Lynxter, anticiper veut dire structurer dès maintenant le code de contrôle de façon modulaire et bien documentée, pour qu'un agent futur puisse s'y brancher — et garder un œil sur les publications Frontier Red Team / robotics d'Anthropic et Google DeepMind.
