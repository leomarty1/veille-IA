Contexte étendu

Cursor 3.9, sorti le 22 juin 2026, introduit une page Customize qui unifie la gestion des plugins, skills, MCPs, subagents, rules, commands et hooks à trois niveaux de scope — user, team et workspace — avec support de MCPs custom. C'est une centralisation : tout ce qui configure le comportement de l'agent passe par une interface unique, scopée selon que la config s'applique à un individu, une équipe ou un workspace.

La page expose une leaderboard des plugins/skills/MCPs les plus utilisés dans l'équipe, ajoutables en un clic — un mécanisme de diffusion virale des bonnes configs en interne. Les plugins gagnent des canvases préconfigurés : des templates de setup partagés réutilisables (ex. Hex Canvas pour la dataviz, Atlassian Canvas pour une vue temps réel issues/projets/docs). Les team marketplaces acceptent désormais l'import de dépôts de plugins depuis GitLab, BitBucket ou Azure DevOps, en complément de GitHub — couvrant les organisations qui ne sont pas sur GitHub.

La logique de packaging converge explicitement avec celle de Claude Code / Cowork (skills + MCP + hooks) : les mêmes briques d'extension, organisées de la même façon. Cette convergence facilite la portabilité d'un workflow agent d'un outil à l'autre — un set de skills + MCP conçu pour Claude Code se transpose plus facilement vers Cursor et inversement. Pas d'annonce officielle fetchable retenue ; faits via classmethod.jp, vibecoder.me et tessl.io.

🛠 Pourquoi c'est utile à savoir pour Lynxter

La convergence du packaging MCP/skills facilite la portabilité de tes workflows agents. Si Lynxter conçoit des workflows agents pour les S300X/S600D (génération de profils, parsing de tickets support, audit firmware) sous forme de skills + MCP + hooks dans Claude Code, le fait que Cursor 3.9 adopte la même structure signifie que ces workflows se transposent plus facilement d'un outil à l'autre. Pour une équipe qui mélange Claude Code et Cursor selon les préférences des devs, c'est un argument de standardisation : packager une fois en briques skills/MCP, déployer sur les deux.

Le scope team/workspace permet de distribuer un set MCP standardisé interne. Le modèle à trois scopes (user/team/workspace) répond directement à un besoin Lynxter : distribuer un set MCP standardisé à toute l'équipe support — connecteurs docs machine, base de calibration, tickets — sans que chaque technicien le configure manuellement. C'est le même bénéfice que les .claude/workflows/ partagés via git côté Claude Code, et la leaderboard ajoute un mécanisme de découverte : les configs les plus utiles remontent et se diffusent en un clic.

Cursor reste un second source à benchmarker, pas un remplaçant. Lynxter est sur Claude Code en stack principal ; Cursor 3.9 ne change pas ce choix. L'intérêt est de noter que l'écosystème agent converge vers un standard de facto de packaging (skills + MCP + hooks à scopes), ce qui réduit le coût de switch et le lock-in — bénéfique pour Lynxter qui garde ainsi une flexibilité d'outillage. À surveiller : si Cursor expose des canvases préconfigurés pertinents pour des workflows industriels, ils pourraient inspirer des templates Lynxter réutilisables côté Claude Code.
