Contexte étendu

Perplexity lance Computer for Counsel le 24 juin 2026, déclinaison verticale de son agent Computer pour les équipes juridiques, sur les plans Enterprise et Max (Max à 200 $/mois). Le cœur architectural est le routage par sous-tâche : l'agent envoie chaque étape (recherche, raisonnement, rédaction de contrats) vers le meilleur des 20+ modèles frontière disponibles, sans lock-in fournisseur. Cette approche model-agnostic contraste frontalement avec Harvey (mono-stack) ou les suites propriétaires de Thomson Reuters.

L'agent s'intègre nativement à Microsoft 365 (rédaction dans Word, fichiers SharePoint, contexte Outlook/Teams), à Google Workspace et à 400+ outils via MCP, plus des sources légales premium (Midpage pour case law + citator, Deel, LegalZoom, NetDocuments, DeepJudge, DocuSign). Perplexity se positionne explicitement comme couche workflow / recherche / rédaction et non comme remplaçant de Westlaw, LexisNexis ou Bloomberg Law — les good-law checks restent délégués à Midpage.

Le cabinet Gunderson Dettmer a annoncé approfondir son usage suite au lancement, signal d'adoption par un cabinet de premier plan. L'angle stratégique de Perplexity : ne pas être un modèle mais un orchestrateur qui route intelligemment vers le meilleur modèle par tâche, le tout branché sur l'écosystème MCP large. Pas d'annonce officielle fetchable retenue (changelog en 403) ; faits via Law.com, LawNext, AboveTheLaw et TechTimes.

🛠 Pourquoi c'est utile à savoir pour Lynxter

L'architecture multi-modèle routée via MCP est le vrai enseignement transposable. Computer for Counsel confirme la maturité d'un pattern : un agent qui route chaque sous-tâche vers le meilleur modèle disponible, branché sur 400+ connecteurs MCP, bat le mono-modèle sur des workflows hétérogènes. Pour Lynxter, ce pattern est pertinent pour des workflows internes hétérogènes (devis, support S300X/S600D, doc technique) où certaines sous-tâches gagnent à un modèle rapide/économique et d'autres à un modèle de raisonnement fort — plutôt que tout passer par un seul modèle. À noter pour le design de tes Routines : le routage par sous-tâche est une optimisation coût/qualité, pas seulement une commodité.

Tempérer : le routage multi-modèle ajoute de la complexité de gouvernance. Router vers 20+ modèles signifie multiplier les fournisseurs, les clés, les périmètres de données — exactement ce que Mistral et Anthropic cherchent à scoper cette semaine. Pour Lynxter, qui a fait le pari Claude comme stack principal, l'intérêt n'est pas de répliquer le routage 20-modèles de Perplexity mais d'en retenir le principe à petite échelle : utiliser le gradient de modèles Claude (Opus 4.8 pour le raisonnement, fast mode pour le volume) et éventuellement un second source on-prem (Mistral) pour les workloads souverains, sans exploser la surface de gouvernance.

Le pattern M365 + MCP est à surveiller pour tes propres connecteurs. L'intégration native Word/SharePoint/Outlook montre où va l'orchestration agent enterprise : l'agent travaille dans les outils où vivent déjà les documents, via MCP. Pour Lynxter, si la doc technique et les devis vivent dans M365 ou Google Workspace, exposer ces sources via MCP à un agent Claude (Claude Tag pour le canal support, Claude Code pour l'automation) suit la même logique — la valeur est dans le branchement MCP sur les outils existants, pas dans une nouvelle interface.
