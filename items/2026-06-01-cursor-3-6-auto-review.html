Contexte étendu

Cursor sort la version 3.6 de son IDE le 29 mai 2026, focalisée sur deux capacités majeures côté agentique. Auto-review est un nouveau run mode qui permet à Cursor de travailler plus longtemps sur des tâches sans demander d'approbation à chaque étape. Le mode s'applique à trois classes de tool calls : Shell (commandes système), MCP (serveurs Model Context Protocol externes), et Fetch (requêtes HTTP/web). Le mécanisme : les appels présents dans une allowlist explicite tournent immédiatement sans prompt ; ceux qui peuvent être sandboxés tournent dans un sandbox isolé ; le reste passe par un classifier subagent qui décide en autonomie d'autoriser, de retenter avec une approche différente, ou de demander l'approbation utilisateur. C'est la même philosophie qu'acceptEdits + bypassPermissions côté Claude Code, raffinée avec une couche d'analyse intermédiaire automatique pour réduire la fatigue des prompts d'approbation sans laisser l'agent en roue libre complète.

Build in Parallel attaque le même problème que les Dynamic Workflows Anthropic mais sous un angle différent. Quand on a un plan multi-step, Cursor analyse les dépendances entre tâches et construit un graphe d'exécution orienté. Les étapes identifiées comme indépendantes (par exemple, "ajouter une route API" et "écrire les tests unitaires existants") sont dispatchées simultanément à des sous-agents async. Les étapes dépendantes restent séquentielles. Cliquer sur "Build in Parallel" déclenche l'analyse et l'exécution — l'utilisateur voit le graphe visualisé dans l'IDE pendant l'exécution. Cursor met aussi à jour Composer 2.5 avec une meilleure stabilité sur les sustained long tasks, et propose une nouvelle quick action "Split PRs" qui découpe automatiquement les changements en plusieurs PRs logiques en se basant sur le contexte chat pour identifier les slices.

Côté business model, Cursor bascule Bugbot en usage-only (suppression des seat fees) : les Teams payent depuis le on-demand spend, les Individuals depuis leur usage inclus. Les admins Enterprise gagnent une granularité accrue sur le contrôle des modèles : allowlist / blocklist par modèle ET par provider — bloquer entièrement Anthropic, ou autoriser uniquement Claude Sonnet, devient possible en un toggle. Côté ecosystème : Cursor in Jira permet d'assigner un work item à Cursor ou mentionner @Cursor en commentaire pour lancer un cloud agent qui peut fix bugs, add features, update tests, investigate. Canvas sharing (artifacts interactifs créés par les agents — reports, dashboards) gagne un partage par live snapshot link. La release positionne Cursor comme l'agentic IDE avec le plus de surface dans la chaîne dev (IDE + Jira + cloud agents + PR review + canvas) — concurrent direct à Claude Code + GitHub Actions, et à Antigravity Desktop + Codex.

🛠 Pourquoi c'est utile à savoir pour Lynxter

Cursor reste sur le radar comme alternative agentic IDE. Lynxter utilise Claude Code en stack principal — Cursor n'est pas dans le set actif. Mais le pattern de release de Cursor (Auto-review + Build in Parallel le 29 mai, soit le lendemain des Dynamic Workflows d'Anthropic) confirme que la convergence multi-agent / parallel execution est désormais standard sur tous les IDE agentic. Les Dynamic Workflows Anthropic et Build in Parallel Cursor résolvent le même problème (paralléliser quand c'est parallélisable) avec deux approches différentes : Anthropic via script JS rerunnable, Cursor via graphe de dépendances analysé à la volée. Pour Lynxter, ça veut dire qu'aucun choix d'IDE actuel ne te ferme une porte sur le multi-agent — la primitive est désormais disponible partout.

Auto-review est un pattern à reproduire dans la configuration Claude Code Lynxter. Le classifier subagent qui décide automatiquement entre allow / retry / ask est plus sophistiqué que le simple acceptEdits de Claude Code. À mettre dans la liste des features à demander en feedback à Anthropic — soit côté Claude Code GitHub Issues, soit en direct via partnership. En attendant, Lynxter peut bricoler un équivalent : configurer un script bash de hook pre-tool-use qui passe les commandes inconnues à un Claude classifier en background, avant d'autoriser ou bloquer. C'est lourd, mais sur les Routines critiques où on ne veut pas tout allowlister mais on veut éviter les prompts incessants, ça vaut un POC.

La bascule Bugbot en usage-only est un signal de pricing à anticiper. Cursor pousse vers le modèle "no seat, pay per use" — Anthropic ne l'a pas (encore) fait sur Claude Code, mais la pression de marché augmente. Pour Lynxter, qui est sur un palier Claude Max actuellement, ça veut dire surveiller les prochaines annonces tarifaires d'Anthropic (rumeurs d'introduction de paliers usage-only enterprise) et ajuster les contrats annuels en conséquence. Si Anthropic introduit un palier "tokens-only enterprise" avant fin 2026, ça peut être plus économique pour Lynxter qu'un palier seat-based — à modéliser dans un tableau avec la conso réelle des Routines actuelles.
