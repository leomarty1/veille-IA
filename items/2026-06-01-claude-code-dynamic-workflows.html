Contexte étendu

Anthropic introduit les Dynamic Workflows dans Claude Code le 28 mai 2026, en research preview, sur tous les plans payants (Pro, Max, Team, Enterprise) ainsi que sur l'API Anthropic, AWS Bedrock, Google Cloud Vertex AI et Microsoft Foundry. Requis : Claude Code v2.1.154+. C'est la troisième brique d'orchestration multi-agent d'Anthropic après les subagents (workers spawned par Claude au tour le tour) et les skills (instructions que Claude suit). Un workflow est différent dans la nature : c'est un script JavaScript que Claude écrit pour ta tâche, et qu'un runtime exécute en arrière-plan dans un environnement isolé du chat. Les résultats intermédiaires vivent dans des variables de script, pas dans le contexte de Claude — c'est ce qui permet de faire tourner des dizaines à des centaines d'agents sans saturer la fenêtre de contexte.

Le runtime applique des limites strictes : jusqu'à 16 agents concurrents (moins sur les machines avec peu de cœurs CPU) pour borner la conso de ressources locale, et 1 000 agents au total par run pour éviter les boucles runaway. Trois façons de déclencher : (1) inclure le mot-clé workflow dans un prompt (Run a workflow to audit every API endpoint under src/routes/ for missing auth checks) — Claude Code highlight le mot et écrit un script à la place de gérer la tâche turn-by-turn ; (2) activer le mode /effort ultracode qui combine reasoning xhigh + orchestration auto — chaque tâche substantielle de la session devient un workflow planifié ; (3) lancer un workflow bundled comme /deep-research qui fan-out web searches, fetch et cross-check sources, et synthétise un rapport cité avec les claims non vérifiés filtrés.

Le workflow tourne en background, la session Claude reste responsive. La commande /workflows ouvre une vue qui affiche chaque phase avec son agent count, son token total et son elapsed time ; on peut drill-in sur chaque agent pour voir son prompt, ses tool calls et son résultat. Le run est pausable et resumable dans la même session — les agents déjà complétés rendent leurs résultats cachés, le reste tourne en live. Sauvegarde : après un run qui a fait ce qu'on voulait, presser s dans /workflows sauvegarde le script comme commande (.claude/workflows/<name> au niveau projet partagé via git, ou ~/.claude/workflows/<name> niveau utilisateur). La commande devient /<name> dans toutes les sessions futures. Permissions : les sous-agents tournent en acceptEdits mode, file edits auto-approuvés ; les shell commands, web fetches et MCP tools hors allowlist peuvent encore prompter mid-run. Exemple flagship cité par Anthropic : Jarred Sumner a porté Bun de Zig à Rust (~750 000 lignes de code Rust générées) en utilisant les Dynamic Workflows.

🎯 Pourquoi c'est important pour Lynxter

Les Dynamic Workflows débloquent ce que les Routines Cowork ne peuvent pas faire aujourd'hui. La Routine actuelle de génération du brief veille-IA tourne en orchestrateur séquentiel : Claude scanne Anthropic, résume, passe à OpenAI, résume, etc. — 12 à 15 minutes wall-clock pour ~30 WebSearch et 10-15 WebFetch. La même tâche en Dynamic Workflow paralllélise : un sous-agent dédié à chaque acteur (8 acteurs principaux + secondaires = 12 sous-agents en parallèle, dans la limite de 16 concurrents), un agent de cross-check qui élimine les doublons, un agent rédacteur final. Le contexte de Claude principal reste libre, seul le rapport final atterrit. Estimation : 3-4 minutes wall-clock total, soit un facteur 4×. À tester cette semaine : activer /effort ultracode dans une session Claude Code 2.1.154+, demander la génération du brief, observer le plan automatique. Si la qualité du résultat est égale ou supérieure, la Routine prochaine peut basculer sur ce pattern et économiser ~10 minutes par exécution hebdomadaire.

Le mot-clé workflow change ton usage CLI quotidien. Désormais, pour toute tâche multi-étapes ambiguë (audit firmware S300X complet, migration de scripts test, recherche cross-fichiers dans la base de calibration), tu n'as plus besoin de structurer manuellement le multi-agent. Tape "workflow trouve tous les profils PEI 90A avec une dérive de température > 5°C sur les 6 derniers mois et propose une correction" — Claude écrit le script (un sous-agent par profil identifié, un agent d'analyse statistique, un agent de proposition correctif), tu approuves le plan, le run tourne en arrière-plan pendant que tu continues sur autre chose. Si le run est utile, sauvegarde-le comme /audit-pei90a dans .claude/workflows/ du repo Lynxter — il devient une commande standard pour toute l'équipe support.

Critère de bascule : sur les 3 prochaines semaines, identifie les Routines Cowork où le wall-clock dépasse 5 minutes et où la tâche a une structure clairement parallélisable (un sous-job par item d'une liste). Bascule-les en Dynamic Workflow et mesure le gain. Pour les Routines < 2 minutes ou avec des dépendances séquentielles fortes (chaque étape dépend de la sortie de la précédente), garde l'orchestration classique — les workflows consomment plus de tokens par run (l'overhead du script + des agents multiples), donc il faut un vrai gain wall-clock pour amortir. Anthropic mentionne explicitement que "un workflow utilise meaningfully plus de tokens que la même tâche en conversation" — à intégrer dans tes monitorings de coût Cowork.
