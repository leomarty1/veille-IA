Contexte étendu

xAI sort Grok 4.5 le 8 juillet 2026, son premier modèle explicitement pensé pour le code et l'agentique, co-entraîné avec Cursor sur des trillions de tokens d'interactions dev/agent. Artificial Analysis le place 4e de son Intelligence Index à 54, derrière Fable 5 (60), Claude Opus 4.8 (56) et GPT-5.5 (55) — un saut de +16 points par rapport à Grok 4.3.

Sur le code, il marque 76 au Coding Agent Index — à parité avec GPT-5.5 (xhigh) sous Codex, juste sous Fable 5 (max) sous Claude Code — et 64,7 % sur SWE-bench Pro. L'argument central est économique : 2 $/6 $ par M tokens jusqu'à 200K de contexte (cached 0,50 $), 4 $/12 $ au-delà, fenêtre de 500K, et environ 4,2× moins de tokens consommés qu'Opus 4.8 pour une tâche facturée ~0,31 $. Le co-entraînement Cursor le rend disponible dans Cursor sur tous les plans.

La contrepartie est sévère et documentée par Artificial Analysis : le taux d'hallucination (AA-Omniscience) bondit de 25 % à 54 % entre Grok 4.3 et 4.5 — le modèle est optimisé pour agir vite et pas cher, au prix de la fiabilité factuelle. Il n'est pas encore disponible dans l'UE.

🛠 Pourquoi c'est utile à savoir pour Lynxter

Grok 4.5 est le pendant xAI de Muse Spark côté « modèle agentique low-cost » : un candidat sérieux sur le rapport coût/exécution (~4,2× moins de tokens qu'Opus 4.8, à parité GPT-5.5 sur le Coding Agent Index), mais avec un défaut disqualifiant pour la plupart des usages Lynxter — un taux d'hallucination de 54 %. Pour un audit firmware S300X/S600D, une migration de scripts ou une réponse support technique, la fiabilité factuelle prime sur le prix : un modèle qui hallucine une réponse sur deux n'est pas utilisable sans double-vérification systématique, ce qui annule le gain de coût.

Deux points d'attention pratiques : (1) l'indisponibilité UE bloque tout usage conforme pour des clients européens sensibles à la localisation ; (2) le co-entraînement avec Cursor signifie que les devs Lynxter utilisant Cursor y ont accès par défaut sur tous les plans — cadre cet accès dans ta politique de gouvernance des agents. À benchmarker éventuellement pour des tâches non-critiques et tolérantes à l'erreur (brouillons, exploration), mais à exclure des workflows où une hallucination coûte cher.
