Contexte étendu

Entre le 8 et le 12 juin 2026, Claude Code enchaîne huit versions (2.1.169 → 2.1.176). La plus structurante est la 2.1.172 (10 juin), qui autorise les sous-agents à spawner leurs propres sous-agents jusqu'à 5 niveaux de profondeur. Jusqu'ici, un sous-agent était une feuille de l'arbre d'orchestration ; il peut maintenant devenir lui-même orchestrateur, ce qui permet de décomposer une tâche en hiérarchies (un agent « acteur » qui pilote des agents « recherche » puis « rédaction », par exemple). La même version ajoute une barre de recherche dans le marketplace /plugin et corrige un bug WebFetch où un domaine wildcard de l'allowlist ne matchait pas les sous-domaines.

La 2.1.169 (8 juin) introduit le flag --safe-mode (et la variable d'environnement CLAUDE_CODE_SAFE_MODE), qui désactive toutes les customisations — skills, plugins, hooks, MCP servers — pour repartir d'un état vierge. S'ajoutent la commande /cd pour déplacer une session de répertoire de travail sans la relancer, un setting disableBundledSkills, et un correctif des policies MCP managées qui n'étaient pas réappliquées à la reconnexion. La 2.1.170 (9 juin) branche Fable 5 dans Claude Code, avec auto-fallback vers Opus 4.8 pour les organisations sans accès au modèle premium.

Les versions 2.1.174/175 (12 juin) apportent les deux leviers de gouvernance. enforceAvailableModels verrouille le modèle Default dans une allowlist managée : les settings utilisateur ou projet ne peuvent plus élargir une liste availableModels définie au niveau de l'organisation. Côté observabilité, l'extension VSCode affiche désormais une attribution d'usage détaillée — cache misses, long context, sous-agents — avec un breakdown par skill, agent, plugin et MCP. Enfin la 2.1.176 génère les titres de session dans la langue de la conversation et ajoute un setting footerLinksRegexes.

🎯 Pourquoi c'est important pour Lynxter

Les sous-agents imbriqués changent le plafond de ce que la routine veille peut orchestrer. Le brief que tu lis fait déjà du fan-out (un agent par acteur). Avec l'imbrication sur 5 niveaux, on peut structurer plus proprement : un orchestrateur top-level → un agent par acteur → des sous-agents recherche/croisement/rédaction par acteur, chacun rendant un résultat consolidé sans saturer le contexte du parent. À combiner avec les Dynamic Workflows : tu gagnes en parallélisme sans perdre la lisibilité du plan. Action : refactore la routine veille pour tester un niveau d'imbrication supplémentaire sur l'acteur le plus dense (Anthropic) et mesure le wall-clock.

enforceAvailableModels et l'attribution d'usage VSCode sont les outils de gouvernance de flotte qui manquaient. Avec enforceAvailableModels, tu imposes Opus 4.8 comme Default à toute l'équipe depuis une config centrale et tu bloques l'usage d'un modèle hors budget (ou instable, comme Fable 5 après sa suspension) sans pouvoir être contourné en local. L'attribution d'usage te donne enfin le coût réel par skill/agent/MCP — indispensable pour identifier quelle routine consomme et où placer le fast mode. --safe-mode complète le tableau pour débugger : quand une routine dérape, relance-la sans aucune customisation pour isoler si le problème vient d'un hook/skill maison. Pose ces trois leviers cette semaine.
