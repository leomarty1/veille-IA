Contexte étendu

Google DeepMind intègre le « computer use » comme outil natif directement dans Gemini 3.5 Flash le 24 juin 2026. Jusqu'ici, le contrôle d'écran n'existait chez Google que sous forme d'un modèle standalone dédié, Gemini 2.5 computer use ; il devient maintenant une capacité de premier ordre du modèle Flash généraliste. Le modèle voit et pilote un écran — web, mobile, desktop — en observant des captures et en émettant des actions (clic, frappe, scroll). Il score 78.4 % sur OSWorld-Verified, le benchmark de référence pour le contrôle d'OS agentique.

Ce 78.4 % place Gemini 3.5 Flash quasi à parité avec GPT-5.5 (78.7 %) et Claude Sonnet 4.6, mais nettement sous Claude Opus 4.8 (83.4 %) et sous l'ex-leader Claude Fable 5 (85.0 %, suspendu le 12 juin par directive d'export-control US et donc indisponible). L'argument de Google n'est pas la performance brute mais le rapport capacité/coût : Flash conserve son positionnement coût/latence, à moins de la moitié du prix de modèles frontière comparables, ce qui en fait le candidat naturel pour des workflows de contrôle d'écran à fort volume où la latence et le coût priment sur le dernier point de précision.

L'intégration native apporte un avantage concret : le contrôle d'écran se combine avec les outils Search et Maps dans un même appel, sans orchestration externe — un agent peut chercher une info, puis agir sur l'écran, dans un seul tour. Disponible via l'API Gemini, AI Studio et Vertex AI. Réserve méthodologique importante : tous les scores OSWorld-Verified restent auto-déclarés par les éditeurs, sans vérification tierce indépendante en juin 2026 — à traiter comme des ordres de grandeur, pas comme des mesures arbitrées. Pas d'annonce officielle fetchable ; faits via The Decoder, eWeek et The Next Web.

🛠 Pourquoi c'est utile à savoir pour Lynxter

Un agent de contrôle d'écran low-cost/low-latency est pertinent pour les bancs de test logiciels Lynxter. Les outils métier autour des S300X/S600D — slicers (PrusaSlicer, Cura), outils de calibration propriétaires, anciens logiciels Windows fournisseurs filament — n'ont souvent pas d'API. Un agent computer use peut automatiser des workflows UI répétitifs : capturer l'état d'un slicer, vérifier une config, exporter un gcode avec un profil donné. Le positionnement coût/latence de Flash le rend candidat pour les tâches à fort volume où la précision absolue compte moins (vérifs de config en masse, captures d'état machine), là où Claude Opus 4.8 (83.4 % OSWorld) reste préférable pour les workflows critiques.

Action de benchmark concrète : prendre un cas d'usage UI répétitif Lynxter — par exemple « ouvre PrusaSlicer, charge ce STL, applique le profil PEI 90A, exporte le gcode, range-le dans le bon dossier » — et l'exécuter sur Gemini 3.5 Flash computer use, Claude Opus 4.8 computer use et Codex Computer Use en parallèle sur 20 itérations. Mesurer taux de réussite, temps wall-clock, coût total et gestion des erreurs UI imprévues. L'écart de 5 points OSWorld entre Flash (78.4 %) et Opus 4.8 (83.4 %) devrait se voir sur les flows UI fragiles ; si Flash tient > 80 % de réussite sur ce cas précis à moitié prix, il devient le candidat pour cette classe de tâches non-critiques.

Garder en tête la réserve sur les scores. Tous les chiffres OSWorld-Verified étant auto-déclarés sans arbitre tiers, ne pas piloter une décision d'outillage sur l'écart de benchmark seul — c'est le test interne sur tes propres workflows qui tranche. Le signal de fond reste que le computer use bascule du statut « modèle dédié » au statut « capacité native intégrée » chez les trois grands (Anthropic, OpenAI, Google) : la primitive est mûre, le différenciant se déplace sur le coût et la fiabilité réelle sur tes tâches, pas sur l'existence de la capacité.
