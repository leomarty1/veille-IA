Contexte étendu

Anthropic enchaîne une nouvelle salve de versions Claude Code du 23 au 26 juin 2026, v2.1.187 à 2.1.195, suite directe aux 2.1.178-185 couvertes le 22 juin. La pièce maîtresse de v2.1.187 (23 juin) est le réglage sandbox.credentials : il bloque la lecture des fichiers de credentials et des variables secrètes par les commandes sandboxées. Un agent qui exécute du shell dans le sandbox ne peut plus lire ~/.aws/credentials, les tokens en variables d'environnement ou les fichiers de secrets — la surface d'exfiltration par un agent compromis ou mal aligné se ferme. La même version ajoute des restrictions de modèle configurées par l'org (appliquées au model picker, à --model, à /model et à ANTHROPIC_MODEL), l'abort des appels MCP distants bloqués >5 min, un fix du mojibake CJK et une réduction de la croissance mémoire en sessions longues.

v2.1.191 (24 juin) introduit /rewind : reprendre une conversation d'avant un /clear — utile quand on a effacé le contexte trop tôt et qu'on veut revenir à l'état antérieur sans tout reconstruire. L'arrêt d'agent depuis le tasks panel devient permanent (un agent stoppé ne redémarre plus tout seul). v2.1.193 (25 juin) durcit l'observabilité et la résilience : les raisons de refus auto-mode sont désormais tracées dans /permissions et le transcript (audit-trail de pourquoi l'agent a refusé une action), la reconnexion MCP OAuth est automatique sur 401/403 (un connecteur dont le token expire se ré-authentifie sans casser la session), une notice au démarrage signale qu'un serveur MCP requiert une auth, et les shells idle voient leur mémoire reapée.

v2.1.195 (26 juin) corrige les hook matchers — les identifiants à tiret comme code-reviewer ou mcp__brave-search sont désormais traités en exact-match, évitant les matches parasites — et ajoute un consentement explicite d'installation pour les plugins externes. L'ensemble de la salve prolonge la trajectoire des semaines précédentes : durcissement sécurité (sandbox, secrets, consentement plugins), résilience des connecteurs MCP (OAuth, timeouts), et observabilité des décisions de l'agent (audit-trail des refus). Source primaire disponible et citable : le changelog officiel code.claude.com.

🎯 Pourquoi c'est important pour Lynxter

sandbox.credentials est le déblocage sécurité pour faire tourner des agents sur les postes support. Si Lynxter exécute des agents Claude Code sur des machines techniciens qui portent aussi des credentials (accès warehouse, tokens API fournisseurs, clés de signature firmware), le risque qu'un agent lise ces secrets via une commande shell était un frein réel. Activer sandbox.credentials ferme cette surface : les commandes sandboxées ne peuvent plus lire les fichiers de secrets ni les variables d'environnement sensibles. Action immédiate : auditer les postes support qui exécutent des agents, activer ce réglage, et vérifier qu'aucune Routine ne dépendait jusqu'ici d'un accès secret en sandbox (auquel cas la faire passer par un connecteur scopé plutôt qu'une lecture directe de fichier).

La reconnexion MCP OAuth automatique fiabilise tes connecteurs maison. Les connecteurs custom branchés aux machines (docs S300X/S600D, base de calibration, tickets) tournent sur des tokens OAuth qui expirent. Avant cette version, un 401/403 en cours de session cassait le connecteur et forçait un redémarrage manuel — inacceptable pour un agent support qui doit tenir une session longue. Désormais la reconnexion est automatique, et la notice de démarrage te signale d'emblée qu'un serveur MCP requiert une auth. C'est exactement la résilience qu'il faut pour passer un connecteur maison du statut « démo » à « production » : combiné aux Access Bundles de Claude Tag et aux clés scopées de Mistral, l'écosystème MCP gagne en robustesse opérationnelle cette semaine.

L'audit-trail des refus auto-mode et les restrictions de modèle par org servent ta gouvernance interne. Le fait que les raisons de refus auto-mode soient tracées dans /permissions et le transcript te donne un journal exploitable : pourquoi l'agent a refusé d'exécuter telle commande, utile pour calibrer tes allowlists et expliquer un comportement à l'équipe. Les restrictions de modèle configurées par l'org permettent de figer quel modèle les techniciens peuvent invoquer (par exemple, forcer Opus 4.8 et interdire un basculement accidentel vers un modèle moins capable sur une tâche firmware critique). Action : si Lynxter a un plan Team/Enterprise, configurer la restriction de modèle au niveau org pour standardiser le backend des agents support, et exploiter l'audit-trail des refus dans tes revues de sécurité.
