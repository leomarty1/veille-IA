Contexte étendu

Anthropic publie Claude Sonnet 5 le 30 juin 2026 et le bascule modèle par défaut de Claude Code dès la version 2.1.197, le jour même. Sur les benchmarks officiels, il atteint 85.2 % sur SWE-bench Verified et 63.2 % sur SWE-bench Pro — en deçà d'Opus 4.8 (88.6 % Verified, 69.2 % Pro) et de GPT-5.5 (88.7 % Verified) sur la performance brute, mais avec un profil agentique nettement plus abouti que le Sonnet précédent : Terminal-Bench 2.1 à 80.4 %, OSWorld-Verified 81.2 %, SWE-bench Multilingual 78.3 %, et BrowseComp à 84.7 % en single-agent (86.6 % en multi-agent).

Le vrai argument est le rapport perf/prix. Le pricing d'introduction est fixé à $2 input / $10 output par M tokens jusqu'au 31 août 2026, puis $3/$15 — soit, même après la promo, environ 40 % du tarif standard d'Opus ($5/$25). Sonnet 5 embarque un contexte natif de 1M tokens et un thinking adaptatif dont l'effort se sélectionne jusqu'à xhigh, ce qui permet d'arbitrer coût et profondeur de raisonnement au niveau de l'appel. Pour un usage agentique où les tokens s'accumulent vite (long context, multi-tours, sous-agents), l'écart de prix avec Opus se paie directement sur la facture mensuelle.

Le timing est coordonné avec le reste de la semaine Anthropic : Sonnet 5 arrive en même temps que la vague de versions Claude Code (subagents en arrière-plan par défaut, Claude in Chrome en GA) et deux jours avant la levée des export controls sur Fable 5. La gamme se lit désormais en trois étages : Sonnet 5 pour le volume agentique (défaut, pas cher, 1M ctx), Opus 4.8 pour les tâches les plus dures (leader SWE-bench Pro), et Fable 5 pour le frontier absolu (95 % Verified, mais quotas bridés post-blackout). Aucun concurrent n'aligne un modèle à ce rapport perf/prix agentique à cette date.

🎯 Pourquoi c'est important pour Lynxter

Sonnet 5 devient le modèle que tes sessions Claude Code utilisent par défaut — sans que tu changes rien. C'est le point d'action prioritaire de la semaine : dès la v2.1.197, toute nouvelle session Claude Code tourne sur Sonnet 5, pas sur Opus. Pour les workflows Lynxter déjà en place (génération du brief veille, scan de tickets support, audit de profils d'impression S300X/S600D), ça signifie un coût divisé par ~2,5 vs Opus pendant la fenêtre promo, au prix d'un léger recul sur les tâches multi-fichiers mal spécifiées (63.2 % Pro vs 69.2 % pour Opus 4.8). À faire cette semaine : relance une routine représentative sur Sonnet 5, compare la qualité subjective à ta baseline Opus, et fixe une règle explicite (« Opus pour les audits firmware, Sonnet 5 pour le reste ») plutôt que de subir le défaut.

La fenêtre de pricing promo est un levère de négociation à exploiter avant fin août. À $2/$10, Sonnet 5 rend économiquement viable ce qui était limite sur Opus : batch de classification de logs, déduplication de base de connaissance, pré-traitement de gros volumes documentaires. Le contexte 1M autorise en plus d'avaler un firmware complet ou une base de profils d'un coup, sans chunking. Si tu envisages une montée en charge du support client assisté par agent au S2, c'est le moment de dimensionner les coûts sur la base Sonnet 5 promo et, si un engagement annuel est sur la table, de figer le tarif avant le passage à $3/$15.

Re-benchmarke avant de basculer les workflows critiques. Le gain de prix ne vaut que si la qualité tient sur tes tâches. Prends deux cas réels — un audit de cohérence de paramètres firmware et une migration de scripts de calibration — exécute-les en parallèle sur Sonnet 5 et Opus 4.8, et mesure l'écart concret (pas le benchmark public). Si la dérive reste sous ~10 %, bascule ces workloads sur Sonnet 5 ; sinon garde Opus là où le coût d'erreur est élevé. Documente la règle dans les settings partagés de l'équipe pour que le choix de modèle ne dépende pas de qui lance la session.
