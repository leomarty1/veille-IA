Contexte étendu

En quatorze releases sur deux semaines, Anthropic industrialise la boucle agentique de Claude Code plutôt que d'ajouter des capacités visibles. La 2.1.207 (11 juil.) est la plus lourde de conséquences pour les déploiements cloud : elle bascule le modèle par défaut sur Claude Opus 4.8 pour Amazon Bedrock, Google Vertex / Agent Platform et Claude Platform on AWS, et surtout active l'auto mode sur ces plateformes sans exiger la variable CLAUDE_CODE_ENABLE_AUTO_MODE qui restait nécessaire ailleurs. Un pipeline CI qui tournait en mode contrôlé sur Bedrock peut donc se retrouver en auto mode par simple montée de version.

La 2.1.212 (17 juil.) refond l'orchestration multi-agent. /fork change de sémantique : au lieu de brancher une variante dans la session courante, il copie la conversation dans une nouvelle session de fond — l'ancien comportement est déplacé sous /subtask. Un cap de 200 spawns de subagents par session (CLAUDE_CODE_MAX_SUBAGENTS_PER_SESSION) borne désormais les workflows fan-out, les appels MCP dépassant 2 minutes passent automatiquement en arrière-plan (CLAUDE_CODE_MCP_AUTO_BACKGROUND_MS), et une limite de 200 WebSearch par session s'ajoute — autant de garde-fous qui prolongent le modèle des Dynamic Workflows (1 000 sous-agents/run) vu chez Anthropic depuis mai.

La 2.1.214 (18 juil.) est une release de durcissement : nouvel outil EndConversation permettant à Claude de clore face à des abus ou tentatives de jailbreak, heartbeat de progression pour les tool calls longs, et plusieurs corrections de permissions notables — le plan mode pouvait lancer des commandes Bash modifiant des fichiers sans prompt, les règles dir/** étaient trop permissives, les commandes de plus de 10 000 caractères passent maintenant par un prompt, et des flags docker daemon-redirect sont corrigés. Enfin la 2.1.215 (19 juil.) met fin à l'exécution automatique de /verify et /code-review, désormais uniquement sur invocation explicite ; la 2.1.200 avait par ailleurs renommé le mode « Default » en « Manual ».

🎯 Pourquoi c'est important pour Lynxter

Action immédiate sur les déploiements : si Lynxter fait tourner Claude Code sur Bedrock, Vertex ou Claude Platform on AWS pour des tâches serveur/CI, audite tes variables d'environnement avant la montée en 2.1.207+. L'auto mode s'active désormais sans opt-in explicite sur ces clouds — ce qui est souhaitable pour des routines batch mais dangereux pour un pipeline qui écrit dans des repos firmware S300X/S600D sans revue. Épingle une version connue et teste le comportement auto mode en environnement isolé avant de propager.

Le changement le plus piégeux est la 2.1.215 : toute routine Cowork ou script maison qui reposait sur l'exécution implicite de /verify ou /code-review ne les lancera plus. Ajoute les appels explicites cette semaine dans tes workflows de génération/migration de scripts métier, sous peine de merges non vérifiés qui passent silencieusement. Dans le même temps, dimensionne tes workflows fan-out (brief veille-IA, audits paramétriques de profils firmware) sous le cap de 200 subagents/session et adapte les scripts qui utilisaient /fork pour du branchement in-session vers /subtask.

À l'inverse, le durcissement des permissions de la 2.1.214 est une bonne nouvelle exploitable dans ton argumentaire de gouvernance : la fermeture du trou « plan mode → Bash mutant sans prompt » et le resserrement des règles dir/** réduisent la surface d'un agent qui déraperait sur des données machine. Combiné à l'outil EndConversation, cela renforce le dossier « Claude Code est le CLI d'agent le plus verrouillé par défaut » — argument directement opposable, cette semaine, à l'incident d'exfiltration Grok Build (voir item xAI).
